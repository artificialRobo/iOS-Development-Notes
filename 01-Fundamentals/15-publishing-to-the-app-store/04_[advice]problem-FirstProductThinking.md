# Notes: Problem-First Product Thinking

## Start with the Problem, Not the Solution

Before thinking about features, MVPs, or even software, you should determine whether the problem actually needs software.

A useful approach is:

### 1. Identify the Problem

Ask:

> What problem am I trying to solve?

Example:

* Students miss assignment deadlines.

At this stage, there is no assumption that software is the answer.

### 2. Understand the Root Cause

Ask:

> Why does this problem happen?

Maybe:

* Students forget deadlines.
* Deadlines are spread across multiple systems.
* Students don't have a reliable reminder system.

### 3. Explore All Possible Solutions

Don't immediately jump to an app.

Possible solutions could be:

* A paper planner.
* A printable checklist.
* An accountability group.
* A text-message reminder service.
* A software application.

The goal is to solve the problem, not necessarily to build software.

### 4. Ask Whether Software Creates a Significant Advantage

Software is often appropriate when the solution benefits from one or more of these:

* Automation
* Real-time updates
* Personalization
* Data storage and retrieval
* Communication between many users
* Scalability

For the deadline example:

* A paper planner can store deadlines.
* Software can automatically send reminders, sync devices, and update schedules.

Software provides additional value that non-software solutions cannot easily provide.

### 5. Validate Demand Before Building

Ask potential users:

> "How are you solving this today?"

If people already have a simple solution they're happy with, software may not be necessary.

If they consistently complain about their current solution, there may be an opportunity.

---

## A Simple Test

Before building software, complete this sentence:

> "This problem is better solved with software because ______."

Good answers:

* "...because it requires automation."
* "...because information changes frequently."
* "...because many people need to collaborate."
* "...because users need access from anywhere."

Weak answers:

* "...because I want to build an app."
* "...because apps are popular."

---

## Key Principle

A startup should start with a **problem**, not a **technology**.

The sequence is:

```text
Problem
↓
Understand the cause
↓
Explore possible solutions
↓
Determine whether software is the best solution
↓
Design the product
↓
Build the MVP
```

If a simple non-software solution solves the problem just as well, building software may add complexity without adding enough value. The best products exist because software is the most effective way to deliver the solution, not because software was the starting point.
